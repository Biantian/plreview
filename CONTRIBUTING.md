# Contributing To PLReview

感谢你愿意为 PLReview 做贡献。

## 开始之前

- 提交较大改动前，先创建 issue 说明目标、范围和预期行为，避免和已有计划冲突。
- Bug 修复、文档修订和小范围可验证改动可以直接发起 PR。
- 本仓库当前以桌面应用为主，优先保证本地优先、离线可用和已有评审流程不回退。

## 本地开发

1. 安装依赖：

   ```bash
   npm install
   ```

2. 复制环境变量模板：

   ```bash
   cp .env.example .env
   ```

3. 初始化数据库：

   ```bash
   npm run db:generate
   npm run db:push
   npm run db:seed
   ```

4. 启动桌面开发环境：

   ```bash
   npm run desktop:dev
   ```

## 提交前检查

请在提交 PR 前尽量完成以下检查：

- 运行 `npm test`
- 如改动影响构建链路或桌面运行时，运行 `npm run desktop:build`
- 如改动影响文档、命令、环境变量或发布流程，更新 `README.md` 或 `docs/` 中对应文档
- 如改动影响用户流程，补充或更新相关测试

## Pull Request 约定

- 保持单个 PR 聚焦一个主题，避免把无关重构和功能改动混在一起。
- 在 PR 描述中说明：
  - 改动背景
  - 主要变更点
  - 验证方式
  - 是否有后续工作未完成
- 如果 UI、交互或桌面打包行为有变化，附上截图、录屏或操作说明。

## 文档与行为边界

- 不把实现说明、设计取舍或发布工程细节直接塞进面向用户的界面文案。
- 这类信息应写在代码注释、`README.md`、`docs/` 或 PR 说明中。
- 若新增功能依赖第三方服务、密钥或平台签名能力，请明确哪些步骤适用于贡献者，哪些步骤仅维护者可执行。
