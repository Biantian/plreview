# 帮助信息收敛与模型配置管理设计

## 1. 背景

当前产品的说明性文案分布在首页、评审列表、新建评审页、详情页等多个页面中。虽然这些文案有助于初次理解流程，但对已经熟悉产品的使用者来说，会持续占用页面空间，削弱“快速进入工作”的体验。

与此同时，项目虽然已经存在 `llm_profiles` 数据表和评审时的模型选择能力，但真实调用链路仍然依赖全局环境变量 `OPENAI_COMPATIBLE_API_KEY`。这使得“按供应商/按配置切换模型连接信息”的能力并未真正建立起来，也无法在页面中安全地管理多套模型配置。

本次设计的目标，是同时解决以下两个问题：

1. 收敛分散的提示教学文本，让产品以熟手工作台为主，而不是每页都重复 onboarding。
2. 提供一个可在页面中维护的模型配置能力，支持手动选择 vendor 与 model，并以适合本机单人使用的方式安全保存 API Key。

## 2. 产品目标

### 2.1 目标

- 首页保留完整工作流程说明，作为唯一主 onboarding 入口。
- 其他业务页面默认保持紧凑，仅保留一句用途说明与按需帮助入口。
- 新增顶部导航入口 `模型设置`，用于集中维护模型 vendor、base URL、默认模型和 API Key。
- 新建评审页只负责“选择配置 + 选择模型”，不再承担配置教学职责。
- API Key 支持随配置切换，并避免以明文形式直接保存到数据库。

### 2.2 非目标

- 第一版不做多人协作、权限隔离或租户级配置管理。
- 第一版不做全厂商专有协议适配，仅优先支持 OpenAI 兼容供应商。
- 第一版不做复杂的文档系统或 CMS，只需要一个轻量帮助页。

## 3. 约束与前提

- 使用场景以本机单人使用为主。
- 当前项目已有 `LlmProfile` 模型，可在此基础上平滑升级，而不是立即拆分为多张新表。
- 当前 LLM 调用逻辑已支持 `baseUrl + model` 组合，天然适合先扩展为多套 OpenAI 兼容配置。
- 项目现有页面已较偏“任务工作台”，因此帮助信息应继续向“默认收敛、按需展开”方向收口。

## 4. 方案对比

### 4.1 方案 A：极简帮助 + 独立模型页

- 首页保留完整流程说明。
- 其他页面大幅删除说明文本。
- 新增独立 `模型设置` 页面。

优点：

- 页面最清爽。
- 信息边界明确。

缺点：

- 某些页面对首次使用者的上下文支持较弱。

### 4.2 方案 B：按需帮助 + 独立 Doc 页 + 独立模型页

- 首页保留完整流程说明。
- 其他页面只保留一句用途说明。
- 全局提供 `帮助` 或 `Docs` 入口，进入独立帮助页。
- 模型配置使用独立页面管理。

优点：

- 熟手不被打扰，新手仍有清晰入口。
- 信息架构清晰，适合后续继续扩展。

缺点：

- 会新增一个帮助页和一个设置页，结构略复杂于极简方案。

### 4.3 方案 C：页面内帮助弹层 + 配置抽屉

- 不做独立帮助页。
- 说明内容主要通过页面内折叠区、弹层或抽屉展示。
- 模型配置也通过弹窗或抽屉处理。

优点：

- 页面跳转少。

缺点：

- 后续内容增长时容易变得拥挤。
- 配置编辑和密钥管理不够稳定、可维护性较差。

### 4.4 推荐方案

选择方案 B：按需帮助 + 独立 Doc 页 + 独立模型页。

原因：

- 最符合“熟手优先、帮助按需展开”的产品方向。
- 能同时满足当前精简页面诉求和后续扩展需求。
- 模型配置作为长期能力，使用独立页面比弹层方案更稳。

## 5. 信息架构设计

### 5.1 首页

- 首页保留唯一完整的流程说明。
- 首页仍承担品牌感、产品说明和主入口作用。
- 首页中的工作流程说明可继续围绕“上传文档 -> 选择规则与模型 -> 后台执行 -> 报告与原文联动”组织。

### 5.2 其他业务页面

以下页面不再放置大段教学文本：

- `评审列表`
- `新建评审`
- `评审详情`
- `规则管理`

这些页面只保留一句页面级说明，强调“这里能做什么”，不再重复介绍完整工作流。

### 5.3 帮助入口

- 在顶部导航中新增 `帮助` 或 `Docs` 入口。
- 进入独立帮助页，集中承载说明信息。

帮助页第一版建议包含三块内容：

- 工作流程说明
- 规则编写说明
- 模型配置说明

### 5.4 页面内按需帮助

- 新建评审页右侧说明卡保留为轻量帮助卡，但默认收起或显著压缩。
- `评审列表`、`详情页` 中原本较长的教学文案收缩为一句说明，不再占用主要阅读区域。
- 页面内如果仍需要局部帮助，优先使用短提示、折叠卡片或帮助入口链接，而不是大块常驻文案。

## 6. 模型配置设计

### 6.1 导航与入口

- 在顶部导航新增 `模型设置`。
- 作为全局配置入口，与首页、评审列表、规则页并列。

### 6.2 页面职责

模型设置页负责统一管理模型连接配置，包括：

- 查看已有配置
- 新增配置
- 编辑配置
- 启用/停用配置
- 删除配置

新建评审页只负责：

- 选择一个模型配置
- 在该配置基础上选择或填写本次模型名称

### 6.3 配置字段

第一版模型配置表单包含：

- 配置名称
- vendor 显示名
- vendor 类型键值
- API 风格
- base URL
- API Key
- 默认模型
- 常用模型列表（可选）
- 启用状态

其中：

- `vendor 类型键值` 第一版仅支持 `openai_compatible`
- `vendor 显示名` 可用于展示 `DashScope`、`OpenAI`、`DeepSeek` 等实际来源
- `常用模型列表` 用于在新建评审页提供下拉预设，但仍允许手动输入模型名

### 6.4 新建评审页交互

- 先选择模型配置。
- 再带出该配置对应的默认模型与常用模型选项。
- 模型输入采用“预设候选 + 手动输入”的组合形态，而不是纯硬编码下拉。

这样能兼顾：

- 不同 OpenAI 兼容供应商模型名经常变化
- 用户仍然希望少输入、快选择

## 7. API Key 安全方案

### 7.1 推荐方案

采用“本机主密钥加密后落库”的方案。

保存方式：

- 数据库不保存明文 API Key
- 仅保存加密后的密文、是否已配置、以及尾号信息
- 使用单独环境变量作为本机主密钥，例如 `APP_ENCRYPTION_KEY`

### 7.2 数据库存储形态

模型配置中新增以下字段：

- `apiKeyEncrypted`
- `apiKeyLast4`
- `hasApiKey`

前端展示规则：

- 页面只展示“已配置 / 未配置”和尾号信息
- 编辑已有配置时不回显明文 Key
- 若需要修改，则重新输入并覆盖

### 7.3 运行时行为

- 评审执行前，根据 `llmProfileId` 读取对应配置
- 服务端在内存中解密 API Key
- 调用结束后不持久保存明文，也不回传前端
- 错误日志中不得输出 Key、Authorization Header 或完整敏感请求体

### 7.4 不采用的方案

不采用“仅环境变量”的原因：

- 无法实现页面中维护多套配置
- 用户体验不满足本次需求

不采用“明文落库”的原因：

- 虽然实现最简单，但后续很难收回
- 对本机应用来说也属于不必要的暴露面

## 8. 数据模型设计

### 8.1 保留并升级 `LlmProfile`

第一版不强制拆新表，直接升级现有 `LlmProfile`：

- 保留现有字段：`name`、`provider`、`apiStyle`、`baseUrl`、`defaultModel`、`enabled`
- 新增字段：`vendorKey`、`apiKeyEncrypted`、`apiKeyLast4`、`hasApiKey`、`modelOptions`

说明：

- `provider` 可继续承担展示名
- `vendorKey` 用于内部逻辑分流
- `modelOptions` 可用字符串或 JSON 形式保存常用模型候选

### 8.2 评审任务快照

为确保历史任务可追溯，`ReviewJob` 建议保留或新增以下快照信息：

- `profileNameSnapshot`
- `providerSnapshot`
- `modelNameSnapshot`
- `baseUrlSnapshot`（可选）

这样即便后续修改了模型配置，历史任务仍能解释“当时实际用的是哪套配置”。

## 9. 数据流设计

1. 用户在 `模型设置` 页创建或编辑一个配置。
2. 服务端使用 `APP_ENCRYPTION_KEY` 对 API Key 加密后落库。
3. 用户在 `新建评审` 页选择某个配置。
4. 页面带出该配置的默认模型与常用模型候选。
5. 用户提交评审任务。
6. 后台执行链路根据 `llmProfileId` 读取配置，并在内存中解密 Key。
7. 请求按 `baseUrl + model + key` 发往对应 OpenAI 兼容接口。
8. 评审结果写回任务与标注数据，同时保留配置快照。

## 10. 错误处理设计

### 10.1 模型设置页

- 如果 `baseUrl` 非法，阻止保存并给出明确提示。
- 如果未填写 API Key，允许保存配置，但状态标记为“未就绪”。
- 已保存配置再次编辑时，不回显原文 Key。

### 10.2 新建评审页

- 若所选配置未启用，不允许发起任务。
- 若所选配置缺少 API Key，不允许进入真实评审流程。
- 页面应明确提示“配置未完成”，而不是以通用报错替代。

### 10.3 演示模式

当前项目是通过“环境变量缺失”隐式切换到 mock 模式。

建议改为显式能力：

- 若要保留演示模式，应作为明确的配置或开关存在
- 不再依赖“缺 Key 自动进入 mock”这类隐式逻辑

这样有两个好处：

- 用户更容易理解当前到底是在跑真实评审还是演示流程
- 模型配置的“未就绪”状态与“演示模式”不再混淆

### 10.4 后台执行

- 解密失败、请求失败、JSON 解析失败时，将任务标记为失败或部分完成
- 错误信息保留可诊断性，但不泄露敏感凭据
- 删除已有历史引用的配置时，不影响历史任务展示，只影响未来能否继续选择该配置

## 11. 测试与验证策略

### 11.1 数据层

- 覆盖 API Key 加密/解密流程
- 覆盖模型配置保存与更新
- 覆盖编辑时不回显明文 Key 的行为

### 11.2 Server Actions

- 新增模型配置
- 编辑模型配置
- 启用/停用模型配置
- 创建评审时读取正确配置

### 11.3 后台任务执行

- 确认不同配置会传入不同的 `baseUrl + model + key`
- 确认 `ReviewJob` 快照字段正确记录
- 确认错误信息不会带出敏感内容

### 11.4 UI 层

- 顶部导航可以进入 `模型设置`
- 新建评审页切换配置时，默认模型与就绪状态同步更新
- 其他页面不再出现大段常驻教学文案
- 帮助入口存在且能够集中查看说明内容

## 12. 实施建议

建议按以下顺序推进：

1. 收敛文案信息架构，确定帮助页和顶部导航结构
2. 扩展 `LlmProfile` 数据模型与迁移
3. 实现 API Key 加密与服务端读写
4. 新增 `模型设置` 页
5. 改造新建评审页的配置选择逻辑
6. 改造后台任务执行链路，按配置读取 Key
7. 将演示模式改为显式开关或显式配置
8. 补齐测试与回归验证

## 13. 验收标准

- 首页是唯一完整流程说明页面。
- 评审列表、新建评审、详情页、规则页默认不再出现大段教学文本。
- 顶部导航存在 `模型设置` 入口。
- 用户可以在页面中新增至少一套 OpenAI 兼容模型配置。
- API Key 不以明文形式保存在数据库中。
- 新建评审时可以按配置切换 vendor/base URL/default model。
- 后台评审实际使用所选配置对应的连接信息。
- 历史任务保留配置与模型快照，可用于回看。
