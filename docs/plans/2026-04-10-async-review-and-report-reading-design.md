# 异步评审与报告阅读体验改造设计

## 1. 背景

当前评审流程在点击“开始评审”后，会在服务端同步执行完整链路，直到文档解析、规则快照、模型调用、标注入库和报告写回全部完成，才跳转到详情页。这带来三个明显问题：

- 启动动作阻塞，用户在提交后无法立即回到任务工作台
- 报告正文仍按纯文本显示，没有发挥 Markdown 报告的结构信息
- “原文结构”区域过于强调技术分块和块级标签，阅读压力较大

本轮改造的目标，是把“启动任务”“查看进度”“消费结果”拆成更清晰的三段体验，同时降低结果页的结构噪音。

## 2. 改造目标

- 点击“开始评审”后立即返回独立的评审列表页
- 新任务在列表页顶部以“评审中”状态出现，并自动刷新状态
- 评审执行迁移为后台链路，前端不再阻塞等待
- 报告正文以完整常用 Markdown 能力渲染
- 原文区按自然正文排版，仅在命中问题的位置显示可点击提示标签

## 3. 信息架构调整

### 3.1 页面分工

- `新建评审页`：仅负责配置和发起评审任务
- `评审列表页`：作为任务中心，展示进行中与历史评审
- `评审详情页`：作为结果消费页，集中展示摘要、报告正文、问题清单与原文命中位置

### 3.2 导航结果

- 点击“开始评审”后，不再跳转到详情页
- 创建成功后统一跳转到独立的 `评审列表页`
- 列表页保留主操作入口，方便继续发起新任务

## 4. 主交互流

### 4.1 创建任务

用户在 `新建评审页` 完成标题、模型、规则和文件选择后提交表单。服务端仅负责：

- 校验输入
- 解析并保存文档
- 创建 `reviewJob`
- 写入初始状态

任务创建成功后立即跳转到评审列表页。

### 4.2 后台执行

实际评审逻辑由独立后台执行入口接手，负责：

- 创建规则版本快照
- 调用模型完成评审
- 过滤并保存标注
- 回写摘要、评分、Markdown 报告和错误信息
- 更新任务终态

前端不等待后台执行结束。

### 4.3 列表页行为

列表页作为任务中心，按创建时间倒序展示评审任务。新创建任务默认排在顶部，并显示进行中状态。

自动刷新策略如下：

- 仅在评审列表页进行轮询
- 仅当页面处于可见状态时轮询
- 仅当存在 `pending` 或 `running` 任务时继续轮询
- 当所有任务结束后自动停止轮询
- 自动刷新只更新状态和任务摘要，不自动跳转详情页

## 5. 状态设计

沿用现有状态枚举，但重新定义前端呈现重点：

- `pending`：任务已创建，等待后台开始
- `running`：正在评审
- `completed`：评审完成且结果完整
- `partial`：评审完成，但部分内容缺失或部分标注未成功落库
- `failed`：评审失败

前端展示策略：

- `pending` 与 `running` 共享“进行中”视觉体系
- `running` 额外展示动态进行中标志，如轻量脉冲点或细环动画
- `completed`、`partial`、`failed` 使用明确状态色区分

## 6. 评审列表页设计

### 6.1 列表信息结构

每个任务项只展示高价值信息：

- 文档标题
- 创建时间
- 模型信息
- 当前状态
- 问题数和评分（仅在任务结束后展示）
- `查看详情` 入口

### 6.2 进行中任务展示

进行中任务需满足以下体验要求：

- 在视觉上可被快速识别
- 不展示空评分或空问题数
- 自动刷新不会改变用户当前浏览位置
- 不在状态变更后强制跳转详情页

## 7. 评审详情页设计

### 7.1 结构调整

详情页职责明确分成三部分：

- 顶部摘要区：状态、模型、时间、评分、问题数、错误信息
- 报告正文区：Markdown 渲染后的正式阅读区
- 联动区：问题清单与正文命中视图

详情页不再承担“等待任务完成”的职责，只面向结果消费。

### 7.2 报告正文渲染

`reportMarkdown` 改为真实 Markdown 渲染，不再以内嵌纯文本方式展示。必须支持完整常用内容：

- 标题
- 段落
- 有序和无序列表
- 引用
- 代码块
- 表格
- 任务列表
- 分割线

版式要求：

- 使用阅读型宽度控制，避免正文过宽
- 长段落自动换行
- 表格和代码块在局部横向滚动，不撑破页面
- 样式层级清晰，但保持后台工具的稳定感

### 7.3 原文区改造

原“原文结构”区域改为更自然的正文命中视图：

- 按原文顺序连续排版
- 不显示块编号
- 不显示“正文”“标题”“列表项”等结构标签
- 仅在命中问题的位置展示可点击标签
- 标签内容优先使用规则名或简短问题标识
- 当前选中问题对应段落使用轻量高亮

该区域的职责从“展示解析结构”转为“为问题提供上下文”。

### 7.4 联动行为

点击正文中的问题标签后：

- 右侧问题清单同步聚焦对应项
- 问题详情区域展示当前问题
- 原文中的对应段落进入高亮态
- 页面仅做局部平滑滚动，不进行强打断式跳转

点击右侧问题项时，也应反向联动正文中的命中位置。

## 8. 错误与空结果处理

### 8.1 失败任务

失败任务需要同时具备状态反馈和原因可追踪性：

- 列表页展示 `失败` 状态
- 详情页顶部展示错误信息
- 用户可从列表页进入详情页查看失败原因

### 8.2 无问题结果

若评审完成但未命中任何问题，应视为有效完成，而不是异常：

- 列表页展示 `已完成`
- 详情页显示正常报告正文
- 问题区展示“当前没有标注问题”的空态说明

## 9. 技术约束与实现建议

### 9.1 后台执行形态

当前项目基于 Next.js Server Actions 和 Prisma。为降低改造风险，优先采用“两段式”结构：

- 第一段：创建任务并立即跳转列表页
- 第二段：通过独立服务端入口执行后台评审

后台执行入口可以是：

- 应用内部可触发的异步任务接口
- 进程内延后执行机制
- 后续再替换成更稳定的任务队列

本轮优先保证交互正确和状态流转清晰，不在本轮强行引入复杂队列基础设施。

### 9.2 Markdown 能力

当前依赖中尚未包含 Markdown 渲染组件。本轮需要补充稳定的 Markdown 渲染方案，并覆盖表格、任务列表等语法支持。

### 9.3 视觉方向

本轮视觉处理应延续现有“精品工作台”方向，但把重点从结构展示转为阅读和任务流感知：

- 列表页强化进行中状态的可感知性
- 正文区强化文档可读性
- 命中提示尽量轻，不制造新的视觉噪音

## 10. 验收标准

- 点击“开始评审”后，可立即跳转到独立评审列表页
- 新任务能在列表页顶部以“评审中”状态出现
- 列表页会自动刷新状态，但不会自动跳转详情页
- 评审完成后，列表项显示评分和问题数
- 报告正文可正确渲染标题、列表、引用、代码块、表格、任务列表和分割线
- 原文区不显示块编号和结构标签
- 只有命中问题的位置才显示可点击提示标签
- 点击问题标签或问题项时，正文与问题详情能双向联动

## 11. 本轮不做

- 实时推送或 WebSocket 状态同步
- 复杂筛选、批量操作或任务分组
- 报告导出能力
- 重新设计现有数据库状态枚举
