# 本地优先桌面交付设计

## 1. 背景

当前项目已经从“跑通评审闭环”的 MVP 阶段，进入面向正式交付的迭代阶段。目标用户不再是假定具备开发能力的内部技术同学，而是不会技术的策划、产品设计、PM 等业务角色。

这带来两个新的约束：

- 交付形态必须接近“下载安装即可使用”的桌面产品
- 数据默认保存在本机，只有大模型调用走网络

在此基础上，本轮还计划新增以下能力：

- 支持读取 Excel 文件进行评审
- 支持一次选择多个文件，并按同一套规则批量发起多条评审任务
- 支持评审列表与规则列表搜索
- 评审列表与规则列表从卡片式浏览升级为管理后台风格的表格视图

## 2. 目标

- 将项目从“本机运行的开发型网页”升级为“可下载安装的本地优先桌面应用”
- 保持所有业务数据、文件解析结果、规则与评审记录默认只落在本地
- 保持“一个文件一条评审任务”的结果消费方式
- 新增“同规则批处理”能力，即一次选择多个文件，共享一次规则和模型配置，最终创建多条独立任务
- 为后续迁移到更原生的实现方案预留结构边界

## 3. 非目标

本轮不做以下内容：

- 服务端集中部署版本
- 多用户、账号体系、权限管理
- 团队共享数据库或云端同步
- WebSocket、复杂任务队列或分布式执行
- Excel 的原始表格可视化还原
- 按 sheet 拆成多份策划案
- 一次批处理中为不同文件配置不同规则
- 直接切换到 Tauri 或原生客户端

## 4. 可选方案

### 4.1 方案 A：继续维持 Web 应用，只补功能

优点：

- 当前改动最少
- 现有 Next.js 页面与 Server Action 复用最高

缺点：

- 对非技术用户仍然存在安装、环境变量、数据库初始化和升级门槛
- “本地桌面产品”的交付感不足
- 后续要迁移到更原生方案时，业务逻辑与 Web 运行时耦合较深

### 4.2 方案 B：Electron 壳 + 本地应用服务层 + 前端展示层

优点：

- 最符合“下载安装即可使用、本地存储、仅模型调用联网”的目标
- 可较多复用现有 React/Next 前端与 Node 生态能力
- Excel 解析、SQLite、Prisma、本地文件访问都可延续现有技术方向
- 只要模块边界切对，后续迁移到 Tauri 或原生方案的成本可控

缺点：

- 需要引入桌面壳、主进程桥接、安装包与更新机制
- 包体和桌面工程复杂度会高于 Web 版

### 4.3 方案 C：直接改为更原生的桌面方案

优点：

- 更轻量、更接近最终原生体验

缺点：

- 当前项目基于 Next.js、Prisma、Node 服务端思路，直接迁移代价过高
- 容易把“正式交付迭代”变成“基础设施重建”

## 5. 推荐方案

推荐采用方案 B：`Electron` 作为第一代桌面壳，围绕“本地应用服务层”做一次定向重构，而不是继续在 Web MVP 上叠功能，也不是直接重写成更原生的客户端。

这是一次“产品化重构”，不是推倒重来。保留当前已有的评审域模型、React 界面能力和本地数据库方向，但把业务入口从 `Server Action` 迁移到可迁移的本地服务层，并重做导入与列表这两条主流程。

## 6. 架构设计

### 6.1 总体分层

应用拆成四层：

1. `Electron Shell`
   - 负责窗口、菜单、文件选择、系统通知、自动更新、应用生命周期
2. `App Core`
   - 负责文件导入、文档解析、规则管理、批量任务创建、任务执行、SQLite 读写、搜索、模型调用
3. `UI Layer`
   - 负责列表、表格、搜索、详情页与交互状态
4. `Persistence + LLM Gateway`
   - 负责本地持久化与唯一允许联网的模型调用边界

### 6.2 关键边界

- React 组件不直接访问文件系统和数据库
- `Electron` API 不直接散落在页面组件中，而是经由薄桥接层暴露给前端
- 新增业务逻辑不再继续堆在 `Next.js Server Action` 中
- 模型调用统一收口到单独网关层，便于超时、重试、错误映射与后续供应商扩展

### 6.3 可迁移性原则

未来若改为 Tauri 或更原生的实现，应只替换“壳层”和桥接层，而尽量复用：

- 本地应用服务层
- 数据模型
- 文档导入与评审逻辑
- 前端大部分页面与组件

## 7. 数据模型设计

### 7.1 文档模型

坚持“一份文件就是一份 document”：

- `docx`
- `md`
- `txt`
- `xlsx`

都统一落成本地文档记录，并进入同一套评审链路。

建议保留并整理 `documents`，至少包含：

- `id`
- `title`
- `filename`
- `fileType`
- `sourceType`
- `rawText`
- `blockCount`
- `createdAt`

`document_blocks` 继续作为主评审基元，供模型评审与详情页命中定位共用。

### 7.2 批处理模型

新增 `review_batches`，用来表示一次“同规则批处理”：

- `id`
- `name`
- `llmProfileSnapshot`
- `modelNameSnapshot`
- `createdAt`

新增 `review_batch_rules`，记录本批次所用规则快照，避免每条任务重复表达同一组公共配置。

### 7.3 任务模型

`review_jobs` 保持“一文件一任务”，但升级为批次下的独立任务：

- `id`
- `batchId`
- `documentId`
- `status`
- `summary`
- `overallScore`
- `errorMessage`
- `createdAt`
- `finishedAt`

`annotations` 继续挂在单个 `review_job` 下，保持详情页和统计逻辑稳定。

### 7.4 数据整理策略

由于当前阶段可以接受清空开发期数据，本轮允许做结构整理和迁移，而不必为 MVP 数据兼容保留过多包袱。

## 8. 文件导入与评审流程

### 8.1 新建评审页升级为导入工作台

新建评审页不再是单文件表单，而是“待处理文件工作台”：

- 支持一次选择多个文件
- 每个文件先进入待评审表格
- 在本地完成轻量预解析与校验
- 用户统一选择本批次规则和模型配置
- 点击一次“开始评审”，创建一个 batch 和多条任务

### 8.2 Excel 导入策略

Excel 在业务上视为“一份完整策划案”，不按 `sheet` 拆分成多份文档。

解析建议：

- 按 workbook 顺序读取每个 sheet
- sheet 名称写入标题块
- 表格内容按行转文本，形成可读 block
- 最终汇总成一份 `rawText + blocks`

目标不是保留原始 Excel 视觉布局，而是保留策划内容语义与结构可读性。

### 8.3 发起评审规则

- 一次批处理中，规则与模型配置统一设置一次
- 一个文件最终只生成一条独立 `review_job`
- 发起前暴露本地校验结果，例如空文件、解析失败、无可用规则等
- 发起后跳转任务列表，由后台异步执行每条评审任务

## 9. 列表与搜索设计

### 9.1 评审列表

评审列表从卡片流升级为后台表格。每行一条任务，建议展示：

- 状态
- 文档标题
- 原文件名
- 文件类型
- 所属批次
- 模型
- 问题数
- 评分
- 创建时间
- 完成时间
- 行末操作

行末操作建议：

- 查看详情
- 重新发起
- 打开原文件位置
- 更多操作

### 9.2 规则列表

规则列表也升级为表格，默认只承担浏览、搜索和入口管理职责。

建议列：

- 启用状态
- 规则名称
- 分类
- 默认严重级别
- 最近更新时间
- 简要说明
- 行末操作

编辑不再采用大面积内联展开，而改为：

- 侧边抽屉
  或
- 独立编辑面板

### 9.3 搜索

第一版搜索采用本地搜索，不依赖网络。

评审列表搜索范围：

- 标题
- 文件名
- 批次名
- 模型名
- 状态文本

规则列表搜索范围：

- 规则名
- 分类
- 描述
- 模板内容

第一版不做复杂检索语法，只保证输入即筛选、结果稳定、响应快速。

## 10. 为什么需要重构

本轮新增能力表面上是四项功能，但本质上有两个系统级变化：

- 导入模式从“单文件单任务”升级为“同规则批处理”
- 列表形态从“卡片式浏览”升级为“表格式管理”

因此不建议只在现有 MVP 结构上继续补丁式开发，而应做一次范围受控的定向重构。重构重点应落在：

- 本地桌面交付形态
- 应用壳与业务逻辑解耦
- 批次与任务模型升级
- 导入工作台重做
- 评审列表与规则列表表格化
- 本地搜索接入

不建议本轮做全面重写，也不建议同时引入过多基础设施升级。

## 11. 分阶段实施建议

### 阶段 1：桌面化基础与业务入口迁移

- 引入 `Electron`
- 建立桥接层
- 把核心业务能力从 `Server Action` 抽到本地服务层

### 阶段 2：数据模型与导入工作台

- 新增 batch 模型
- 支持批量文件导入
- 增加 Excel 解析

### 阶段 3：任务中心与规则中心重构

- 评审列表表格化
- 规则列表表格化
- 增加本地搜索

### 阶段 4：桌面交付完善

- 本地数据目录与配置管理
- 安装包构建
- 升级与异常恢复体验

## 12. 验收标准

- 用户可通过下载安装包启动应用，而不是依赖命令行和开发环境
- 所有规则、文档、任务、评审结果默认保存在本机
- 只有模型调用走网络
- 支持导入 `docx`、`md`、`txt`、`xlsx`
- 支持一次选择多个文件，并按同一套规则和模型配置发起多条独立任务
- Excel 被视为一份完整策划案，而不是按 sheet 拆任务
- 评审列表与规则列表均以表格方式展示
- 两个列表都支持本地搜索
- 行末操作按钮可进入详情、编辑或其他操作
- 当前结构允许后续替换桌面壳，而不需要推翻业务核心层
