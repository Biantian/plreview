# Adaptive Form Overlay Design

Date: 2026-04-24
Branch: `main`

## Goal

修复“新增模型”和“新增规则”长表单超出应用窗口且无法滚动的问题，并把这两个入口统一到一套可复用的长表单浮层交互模型中。

本轮目标不是只补一个滚动条，而是建立一套稳定的自适应编辑容器：在桌面宽窗下保持右侧抽屉上下文，在窗口偏窄或偏矮时自动切换为居中弹层，同时始终保持头部固定、主体滚动、底部操作固定。

## Scope

### In Scope

- 为模型编辑与规则编辑引入统一的长表单浮层壳体
- 保留现有模型表单与规则表单的业务字段、校验和保存接口
- 支持同一套表单在右侧抽屉与居中弹层之间自适应切换
- 让弹层主体成为唯一滚动容器，避免背景页面接管滚动
- 补充相关组件测试，覆盖打开、关闭、形态切换和表单状态保持

### Out of Scope

- 新增或删减模型、规则的业务字段
- 引入未保存变更确认弹窗
- 调整模型页和规则页整体信息架构
- 扩展到其它编辑场景或全局浮层系统
- 处理端到端动画效果或视觉微调规范

## Problem Summary

当前 [components/model-editor-drawer.tsx](../../../components/model-editor-drawer.tsx) 和 [components/rule-editor-drawer.tsx](../../../components/rule-editor-drawer.tsx) 作为普通页面块挂在管理列表下方，受页面和父容器多层 `overflow: hidden` 约束影响，长表单在较小窗口内会超出可视范围，用户无法继续下滑查看剩余字段和底部操作区。

这带来三个直接问题：

- 长表单可用性失效，影响新建模型与新建规则主路径
- 滚动责任不清晰，页面背景、列表区、编辑区之间相互抢占
- 模型编辑和规则编辑虽然都叫“抽屉”，但实际上没有形成稳定的浮层交互约束

## User Decisions Captured

- 这轮目标选择“完整收口交互模型”，不是只做局部可滚动修补
- 编辑容器采用自适应方案，不强制固定为单一抽屉或单一对话框
- 长表单超出时只允许弹层内部滚动，不让整页负责滚动
- 当窗口尺寸变化导致形态切换时，已填写内容必须保留，切换应尽量平滑

## Approaches Considered

### Approach A: Adaptive Overlay Shell

推荐方案。抽象一个通用长表单浮层壳体，根据窗口宽高在“右侧抽屉”和“居中弹层”之间切换；两种形态复用同一套表单内容和状态。

优点：

- 同时兼顾桌面宽窗上下文和小窗口可用性
- 可以沉淀成模型与规则共享的编辑容器
- 能从根本上统一滚动、关闭、焦点与状态保持行为

代价：

- 需要把现有“内嵌块级抽屉”升级成真正的浮层组件
- 需要补齐形态切换时的状态与焦点处理

### Approach B: Fixed Right Drawer Only

保留现有右侧抽屉形式，只补齐滚动和 sticky 分区。

优点：

- 改动最小
- 最贴近当前页面结构

不足：

- 对低高度或窄窗口的适配能力弱
- 仍然把交互模型绑死在单一布局上

### Approach C: Standalone Editor Layer

把编辑流程提升为更强的独立编辑层或近似全屏编辑体验。

优点：

- 对未来超长、分段、复杂表单扩展性最好

不足：

- 明显超出本轮“修复两个入口并统一长表单交互”的范围
- 会弱化当前列表上下文，改动面偏大

## Recommended Direction

采用 Approach A：引入统一的自适应长表单浮层壳体。这个方案既能解决当前 bug，又能把模型与规则的编辑入口收口到相同的可维护模式，不需要为了这轮问题把交互升级成更重的独立工作区。

## Architecture

### Component Split

建议把当前实现拆成两层：

1. 通用浮层壳体组件
   - 暂定命名为 `AdaptiveFormOverlay` 或语义等价名称
   - 负责遮罩、定位、尺寸判断、滚动边界、关闭行为、焦点管理和语义属性
2. 业务表单内容组件
   - 模型编辑表单继续服务 `ModelManager`
   - 规则编辑表单继续服务 `RulesTable`
   - 负责字段渲染、表单提交、错误提示和业务 payload 组装

这意味着“壳体决定怎么展示”，“表单决定填什么和怎么保存”，避免模型表单和规则表单重复实现浮层框架逻辑。

### Rendering Model

- 打开时渲染真正的浮层容器，而不是继续把内容作为管理页普通文档流的一部分
- 浮层容器覆盖在管理页之上，背景保留上下文但不可交互
- 模型编辑与规则编辑共用同一套壳体 API，只传入各自标题、描述、关闭动作、保存区和表单内容

## Layout And Scrolling Rules

### Three-Zone Structure

浮层内部固定为三段结构：

- `header`
  - 标题、可选描述、关闭按钮
- `body`
  - 表单字段区域
  - 唯一允许 `overflow-y: auto` 的区域
- `footer`
  - 主按钮、次按钮、错误提示、保存中状态

### Scroll Ownership

滚动责任明确如下：

- 只有 `body` 区域允许纵向滚动
- `header` 和 `footer` 始终固定，不随内容滚动
- 背景页面在弹层打开后滚动锁定
- 不允许通过撑开父容器高度来间接实现表单可见

这样可以保证用户永远只需要理解一种滚动方式：滚中间表单内容，而不是滚整个页面或在多层容器之间试错。

### Size Constraints

- 右侧抽屉形态：
  - 使用固定右侧停靠
  - 宽度受上限约束，避免吞掉整个工作区
  - 高度跟随窗口，但与视口顶部和底部保留安全边距
- 居中弹层形态：
  - 居中展示
  - 宽度和最大高度均受约束
  - 超出高度后继续由 `body` 负责滚动

两种形态共享同一套内部三区结构，不复制表单实现。

## Adaptive Behavior

### Switch Conditions

壳体根据窗口条件在两种形态间切换：

- 宽度充足且高度充足时使用右侧抽屉
- 窗口较窄或较矮时切换为居中弹层

本轮不把具体阈值写死在 spec 中，但要求实现时集中在壳体内部维护一处判断逻辑，避免模型页和规则页各自硬编码。

### State Preservation

形态切换只改变外层布局，不卸载表单实例：

- 已输入文本保留
- 已选择选项、勾选状态保留
- 当前错误信息与 busy 状态保留
- 打开中的编辑会话保持连续，不因为切换抽屉/弹层而重置

为此应避免：

- 用不同 key 迫使同一表单在两种形态间重新挂载
- 为抽屉版和弹层版维护两套独立表单组件

## Interaction Behavior

### Open

- 点击“新增模型”“编辑模型”“新增规则”“编辑规则”后打开浮层
- 焦点初始落点应在标题区域或首个可编辑字段，保持键盘可达
- 打开后背景内容不再可交互

### Close

支持以下关闭路径：

- 点击关闭按钮
- 点击取消按钮
- 按下 `Esc`

本轮不增加未保存变更确认，但壳体接口应预留未来接入该能力的空间。

### Save

- 保存按钮放在固定底部操作区
- 保存中状态与错误提示与现有行为保持一致
- 模型与规则仍沿用各自现有保存函数和成功回写逻辑

## Accessibility

- 保持 `role="dialog"` 和 `aria-modal="true"`
- 标题通过 `aria-labelledby` 关联
- 若存在描述文案，通过 `aria-describedby` 关联
- 打开后焦点进入浮层上下文
- 关闭后焦点返回打开前的触发按钮或等价触发位置

## Testing Strategy

### Component Tests

为模型管理和规则管理补充或调整测试，覆盖以下行为：

- 点击新增/编辑入口后打开统一浮层壳体
- 浮层存在稳定的 `header`、`body`、`footer` 结构
- 表单字段渲染在可滚动主体区域
- 在不同窗口条件下能切换为不同形态标记
- 形态切换后已输入的表单值保留
- 关闭按钮、取消按钮和 `Esc` 都可关闭浮层
- 保存中和错误提示在两种形态下都能正确透传

### What Not To Test In This Round

- 动画细节或过渡时长
- 像素级视觉对齐
- 端到端滚动截图验证
- 未保存变更确认逻辑

## Risks

- 如果继续让浮层参与页面普通布局流，滚动 bug 会以别的形式复发
- 如果形态切换通过重新挂载组件实现，会破坏用户填写中的上下文
- 如果背景滚动锁定处理不完整，用户仍可能在表单滚动时误滚动列表页

## Success Criteria

- 在常见桌面窗口尺寸下，新增模型和新增规则表单都能完整访问到底部操作区
- 表单超长时，用户只需要滚动浮层主体区域
- 窗口从宽变窄或从高变矮时，编辑容器能切换形态且不丢失已填写内容
- 模型与规则两类编辑入口复用同一套长表单浮层交互模型
