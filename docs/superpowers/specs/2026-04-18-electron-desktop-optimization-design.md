# Electron 桌面端包体与性能优化设计

日期：2026-04-18

## 1. 背景

当前项目已经完成从 Web MVP 到本地优先桌面应用的第一阶段落地，现有桌面方案基于：

- `Electron` 作为桌面壳
- `Next.js + React` 作为渲染层
- `Prisma + SQLite` 作为本地数据层
- `desktop/core` 承接本地导入、评审批处理、查询与导出等业务逻辑

现状说明当前路线已经具备可用性，但还没有进入“面向正式交付优化”的阶段。主要问题集中在三类：

- 打包产物体积偏大，影响下载、分发和升级体验
- 冷启动链路过重，桌面壳启动速度和首窗口可交互时间不理想
- 运行期重任务与壳层边界不够清晰，导入、解析、批处理等能力容易拖累主进程稳定性

本轮不是重新选择桌面技术路线，也不切换到 `Tauri` 或其它桌面框架，而是在保留 `Electron` 的前提下，对桌面架构做一次系统级优化。

## 2. 目标与非目标

### 2.1 目标

- 保留 `Electron` 作为长期桌面壳方案
- 同时改善安装包体积、冷启动速度和运行期稳定性
- 把当前桌面工程收口为清晰的三层结构：
  - `Electron Shell`
  - `Background Worker Layer`
  - `UI Layer`
- 将数据库访问、文档解析、批量评审、导出等重任务从主进程启动链中拆出
- 建立桌面端性能与包体的可观测基线，避免后续优化只能凭体感判断
- 为后续继续扩展桌面能力保留边界，而不要求本轮重写现有 UI

### 2.2 非目标

- 不迁移到 `Tauri`、`Wails`、`Flutter Desktop` 或真正原生客户端
- 不重写 `React/Next.js` 渲染层
- 不更换数据库、ORM 或大模型调用协议
- 不重做业务流程、页面信息架构或已有评审模型
- 不在本轮引入服务端集中部署或云同步
- 不把本轮优化扩展为“新桌面平台从零重建”

## 3. 现状结论

### 3.1 结构现状

当前桌面端已经具备：

- 主进程入口 `electron/main.ts`
- preload 与 bridge
- `desktop/core` 业务逻辑
- `electron-builder.yml` 打包配置
- 本地优先桌面模式与 Win11 烟测文档

这说明项目已经跨过“是否做桌面版”的阶段，当前真正的问题不是技术路线缺失，而是桌面工程仍处于第一代可用实现，缺少专门面向性能与分发的结构治理。

### 3.2 性能与包体问题判断

结合当前代码结构，可以明确得到以下结论：

- 主进程启动时引入了较多业务模块，导致“显示窗口”与“准备所有重能力”绑定在一起
- 文档解析链路会涉及 `xlsx`、`mammoth` 等重依赖，不适合被主进程在冷启动阶段提前装载
- 打包配置当前以“能正确打包”为主，尚未收敛到“只打必须内容”的优化状态
- 桌面核心能力虽已抽到 `desktop/core`，但仍需要进一步定义哪些逻辑属于壳层、哪些逻辑必须迁到后台工作层

因此，本轮优化的核心不是继续微调单点，而是先完成职责拆分，再在新结构上做打包和启动链治理。

## 4. 可选方案

### 4.1 方案 A：战术优化

只在现有结构上做打包裁剪、压缩配置和少量懒加载。

优点：

- 变更最小
- 能较快拿到一轮体积与启动收益

缺点：

- 主进程仍会继续背负过多业务职责
- 结构问题未解决，后续新增能力时容易反弹

### 4.2 方案 B：结构化优化

保留 `Electron`，但把桌面端真正收口为“壳层 + 后台工作层 + UI 层”，同步推进主进程瘦身、后台进程化、打包治理和可观测性建设。

优点：

- 同时覆盖包体、启动、运行稳定性三个目标
- 更符合桌面长期演进需求
- 可在不重写 UI 的前提下获得结构收益

缺点：

- 改动明显大于单纯打包优化
- 需要明确 IPC、进程生命周期和错误恢复策略

### 4.3 方案 C：激进平台化

继续保留 `Electron` 壳，但把业务核心完全抽成独立本地服务，桌面壳仅作为纯前端容器。

优点：

- 长期边界最干净

缺点：

- 本轮成本过高
- 容易把优化项目扩展成新的平台工程

## 5. 推荐方案

推荐采用方案 B：`Electron` 结构化优化。

原因如下：

- 当前仓库已经有稳定的桌面基础，不需要推翻重来
- 用户本轮明确接受较大调整，但目标仍然是“优化交付”，不是“重写客户端”
- 包体、冷启动、运行期稳定性三件事在现状中共享同一个根因：主进程过重、后台职责不清、打包边界不严
- 只有先把结构收口，后续每一项性能优化才有可持续性

因此，本轮应把 `Electron` 从“可用桌面壳”升级为“可优化、可治理、可持续扩展的桌面壳”。

## 6. 架构设计

### 6.1 总体分层

桌面端收口为三层：

1. `Electron Shell`
   - 负责窗口、菜单、原生对话框、应用生命周期、IPC 网关、后台进程管理
2. `Background Worker Layer`
   - 负责数据库访问、文档导入、批量建任务、评审执行调度、导出、查询服务
3. `UI Layer`
   - 负责页面展示、交互状态、用户操作编排，仅通过 bridge 与后台通信

### 6.2 分层原则

- 主进程不直接承接数据库访问、文档解析和批量评审执行
- 重依赖模块默认不进入主进程启动链
- 前端页面不直接耦合桌面底层实现
- 后台工作层对外暴露稳定的消息协议，而不是把底层模块细节泄漏到 UI
- 新增桌面能力默认先进入后台工作层，只有确需与壳层绑定时才允许进入主进程

### 6.3 进程模型

建议采用：

- `1` 个主进程
- `1` 个长期后台工作进程
- `0~N` 个按需拉起的短生命周期任务进程

职责划分如下：

#### Main Process

- `BrowserWindow` 创建与管理
- 原生菜单与系统生命周期
- 文件选择、目录选择、保存对话框
- preload/bridge 注入
- IPC 路由与后台工作进程托管

#### Background Worker Process

- `Prisma` 初始化与生命周期管理
- 评审任务创建、查询、删除、导出
- 规则查询与搜索
- 导入结果持久化
- 批处理执行调度

#### Task Process

- 大文件解析
- `xlsx` / `mammoth` 等重依赖驱动的转换
- 高 CPU 或高内存峰值的批量评审执行

### 6.4 调用链

用户交互链路收口为：

1. `UI` 发起动作
2. `Main` 只做转发和权限边界控制
3. `Background Worker` 接收请求并决定是否自行处理或分发给 `Task Process`
4. 执行进度和结果通过事件或明确响应返回 `UI`

该链路的关键变化是：窗口显示和基础交互不再等待所有业务模块完成初始化。

## 7. 模块迁移设计

### 7.1 保留在主进程的内容

- 窗口创建与窗口状态管理
- 系统菜单
- 原生对话框
- 应用关闭、激活、单实例等生命周期逻辑
- 背景工作进程启动、重启、心跳、异常恢复

### 7.2 迁移到后台工作层的内容

- `Prisma` 客户端初始化
- 评审任务批量创建
- 评审列表与规则列表查询
- 导入结果写入本地数据库
- 导出清单与导出报告
- 评审执行调度

### 7.3 按需加载或迁移到任务进程的内容

- `xlsx` 解析
- `mammoth` 文档解析
- 大型文档文本抽取
- 单次高耗时评审执行
- 批量导出中的内容聚合步骤

## 8. 打包与产物治理

### 8.1 目标

- 安装包仅包含运行时必需内容
- 建立按模块的体积观测能力
- 减少壳层与后台层的冗余文件携带

### 8.2 策略

- 重写 `electron-builder` 的文件白名单，而不是延续宽松收录
- 明确区分开发依赖、构建依赖和运行依赖
- 校验 `asar`、压缩和平台目标配置是否符合分发需求
- 让每次桌面构建都输出体积清单，能够定位产物增长来源
- 将壳层产物、后台产物、渲染层产物在构建与打包流程上做清晰分组

### 8.3 本轮不追求的事项

- 不在本轮引入复杂的增量更新体系
- 不为了追求极限体积而牺牲开发体验
- 不把打包治理扩展成多平台完全不同的构建系统

## 9. 启动与运行期优化策略

### 9.1 冷启动

冷启动优化优先级应为：

1. 主进程去业务化
2. 首窗口尽快显示
3. 后台工作能力异步就绪
4. 用户首次触发重操作时再加载对应模块

### 9.2 运行期

运行期优化重点不在“把所有操作都做快”，而在于：

- UI 不因主进程被阻塞而卡顿
- 后台重任务具备可控生命周期
- 失败时有明确错误映射和恢复策略
- 后续新增能力不会再次把重逻辑堆回主进程

## 10. 可观测性与验收指标

### 10.1 基线观测

第一阶段必须补齐以下指标采集：

- 冷启动开始时间
- 首窗口可见时间
- 首窗口可交互时间
- 后台工作进程就绪时间
- 文档导入耗时
- 单批评审执行耗时
- 导出耗时
- 打包产物总大小
- 分模块体积占比

### 10.2 第一轮验收目标

建议以相对指标作为第一轮目标：

- 安装包体积较当前基线下降 `20%~35%`
- 冷启动首窗口可见时间下降 `30%+`
- 导入、批量评审、导出期间 UI 不出现明显主进程卡顿
- 主进程源码不再直接依赖 `Prisma` 业务入口、`xlsx`、`mammoth`
- 构建流程可以输出可读的体积报告

## 11. 分阶段实施策略

### 11.1 阶段一：建立测量基线

- 为桌面启动链、后台就绪、重任务耗时和产物体积建立统一观测
- 明确当前基线，避免后续只能凭主观感受判断成效

### 11.2 阶段二：主进程瘦身

- 收口主进程职责
- 去掉主进程对重业务模块的直接依赖
- 让窗口显示不再等待后台全量能力

### 11.3 阶段三：后台工作层拆分

- 新增后台进程入口
- 将数据库访问、导入、查询、导出、任务调度迁入后台工作层
- 将最重的解析和执行能力改为按需加载或下沉到任务进程

### 11.4 阶段四：打包治理

- 精简打包白名单
- 输出体积报告
- 校验平台产物结构
- 持续收敛运行时依赖集合

## 12. 风险与控制策略

### 12.1 主要风险

- IPC 边界不清，导致主进程问题转化为跨进程混乱
- `Prisma` 在后台进程中的生命周期处理不当
- 错误恢复策略不足，导致后台进程异常后桌面端进入半失效状态
- Next 渲染层与桌面后台边界不稳，后续再次把逻辑写回壳层
- 缺少量化基线，导致收益难以评估

### 12.2 控制策略

- 先定义消息协议，再迁移模块
- 对后台进程建立启动、异常、重启和健康检查策略
- 以阶段化方式推进，每阶段都有独立验证点
- 以“禁止主进程承接重业务逻辑”作为工程约束写入实施计划

## 13. 成功标准

- `Electron` 路线继续保留，但桌面端结构明显更清晰
- 主进程收口为壳层，不再承担批量业务逻辑和重依赖加载
- 包体、冷启动、运行期稳定性三项指标都能拿到明确改善
- 后续新增桌面能力有统一归属规则，不再继续侵入主进程
- 第一轮优化完成后，可以在此基础上继续写更细的性能与分发迭代计划

## 14. 延后事项

- 自动更新策略细化
- 崩溃采集与更完整的桌面诊断体系
- 更激进的资源拆包与增量分发
- 若后续确有必要，再评估是否进入更彻底的平台化拆分
