# 评审列表批量删除与批量导出设计

日期：2026-04-17

## 1. 背景

当前 `评审列表` 页面已经支持：

- 表格式展示评审任务
- 按标题、文件名、批次、模型、状态搜索
- 刷新列表状态
- 进入单条任务详情页

但它仍然缺少面向“任务中心”场景的批量操作能力。对于已经积累较多评审任务的桌面版使用者来说，以下动作会变得频繁：

- 清理已无保留价值的历史任务
- 一次导出本轮评审任务清单给团队同步
- 一次导出多条任务的完整报告用于归档或外发

本设计为 `评审列表` 增加：

- 批量删除
- 批量导出清单
- 批量导出完整报告

目标是在不破坏当前表格简洁性的前提下，把批量操作做成明确、安全、可理解的桌面工作流。

## 2. 目标与非目标

### 2.1 目标

- 在 `评审列表` 中支持按勾选行执行批量删除、批量导出
- 支持把“当前筛选结果”一键纳入批量范围，而不要求用户逐行勾选
- 删除前统一要求二次确认，并对“包含进行中任务”的情况使用更强提示
- 把导出拆成两个动作：
  - 导出清单
  - 导出报告
- 导出报告时自动跳过未完成且无报告正文的任务，并向用户反馈跳过结果
- 尽量复用现有表格、数据查询和详情数据结构，避免引入大范围重构

### 2.2 非目标

- 不在本轮加入单行删除按钮
- 不在本轮加入分页能力
- 不在本轮加入导出格式选择器
- 不在本轮加入导出任务中心、后台导出队列或下载历史
- 不在本轮导出原始文档、标注 JSON 或混合归档包
- 不在本轮处理“删除任务时是否一并删除文档记录”的额外策略，默认只删除任务及其标注

## 3. 现状约束

### 3.1 页面与交互现状

- `app/reviews/page.tsx` 使用 `ReviewJobsTable` 呈现任务表格
- `components/review-jobs-table.tsx` 当前负责：
  - 客户端搜索过滤
  - 自动刷新与手动刷新
  - 行级“查看详情”
- 当前表格没有选择态，也没有批量操作入口

### 3.2 数据与持久化现状

- `ReviewJob` 通过 Prisma 持久化
- `Annotation.reviewJobId` 对 `ReviewJob` 使用 `onDelete: Cascade`
- `ReviewJob.documentId` 指向 `Document`，但删除 `ReviewJob` 不会自动删除 `Document`
- 详情页已能展示：
  - `summary`
  - `overallScore`
  - `reportMarkdown`
  - `annotations`

这意味着本轮可以安全支持“删除评审任务”，且不会误删原始文档记录。

## 4. 用户需求结论

在本轮脑暴中已经确认：

- 需要两种导出能力同时存在：
  - 导出表格清单
  - 导出完整评审报告
- 删除规则：
  - 所有任务都允许删除
  - 所有删除都需要二次确认
  - 若所选任务中包含 `pending` 或 `running`，提示语必须更强
- 批量操作范围：
  - 支持基于手动勾选行操作
  - 支持对“当前筛选结果全部”生效
- 导出报告规则：
  - 允许批量发起
  - 自动跳过未完成任务
  - 操作完成后反馈导出数与跳过数

## 5. 交互设计

### 5.1 总体方案

采用“选择态批量操作条”方案。

默认状态下，`评审列表` 页面尽量保持当前简洁结构，只展示：

- 搜索框
- 简短说明
- 刷新按钮

当用户勾选任意一条任务后，在表格上方插入一条批量操作条，显示：

- 已选数量
- 当前筛选结果全部选择入口
- 清除选择
- 导出清单
- 导出报告
- 删除

这样做的原因是：

- 不把危险操作常驻在默认视图里
- 在视觉上把“浏览列表”与“批量操作”分成两种上下文
- 为后续追加更多批量动作保留空间，而不会挤压默认工具栏

### 5.2 表格选择模型

表格新增一列复选框：

- 表头复选框：作用于“当前筛选结果”
- 行复选框：作用于单条记录

选择状态分为两层：

1. `selectedIds`
   - 用户手动勾选的任务 id 集合
2. `allMatchingSelected`
   - 当前筛选结果全部选中状态

当用户点击“选择当前筛选结果全部 M 项”时，不要求前端在语义上把每条记录都转换成手工勾选；应显式进入“当前筛选结果已选中”状态，并允许用户再点击“仅保留手动勾选”或“清除选择”退出。

### 5.3 批量操作条文案

文案整体保持短句，不写长解释。

建议文案方向：

- `已选 3 项`
- `全选筛选结果 18 项`
- `仅保留已勾选`
- `清除选择`
- `导出清单`
- `导出报告`
- `删除`

若进入“当前筛选结果全部”状态，可切换为：

- `已选筛选结果 18 项`
- `仅保留已勾选`
- `清除选择`

不再额外附加长段说明，避免占据工具栏空间。

### 5.4 筛选与选择的联动

搜索词变化时，筛选结果会变化，因此需要明确联动规则：

- 默认只保留仍然存在于结果集中的 `selectedIds`
- 如果处于 `allMatchingSelected` 状态，搜索词变化后，选择范围自动切换到新的筛选结果集合
- 批量条中的数量展示始终反映“当前实际作用范围”

这样可保证“对当前筛选结果全部生效”的语义保持一致。

### 5.5 空状态与无匹配状态

- 当列表本身为空时，仍保持现有空状态，不显示选择态 UI
- 当搜索后无匹配结果时，不显示批量条
- 若用户此前已有选择，但当前搜索导致结果为 0，选择态自动清空

## 6. 删除设计

### 6.1 删除范围

删除动作支持两种来源：

- 手动勾选的任务
- 当前筛选结果全部

最终删除集合由统一服务在服务端解析，避免前端和后端对“范围”理解不一致。

### 6.2 二次确认

所有删除都要求二次确认，但根据任务状态切换提示强度。

#### 普通确认

适用于选择集合中不包含 `pending` 或 `running`。

建议文案：

- 标题：`确认删除`
- 正文：`删除后无法恢复，确认删除这 3 条评审任务？`

#### 强确认

适用于选择集合中包含 `pending` 或 `running`。

建议文案：

- 标题：`确认删除进行中任务`
- 正文：`所选任务中包含进行中的评审，删除后无法恢复，确认继续删除？`
- 辅助摘要：`包含 2 条进行中、4 条其他任务`

若选择集合为混合状态，只弹一次更强确认，不拆成两次确认。

### 6.3 删除影响范围

删除 `ReviewJob` 时：

- 级联删除其 `Annotation`
- 保留 `Document`
- 保留 `ReviewBatch`

本轮不尝试做“孤儿文档回收”，以免扩大作用范围。

## 7. 导出设计

### 7.1 导出清单

`导出清单` 永远允许执行，不区分任务状态。

默认导出格式为 `xlsx`，字段包括：

- 标题
- 文件名
- 文件类型
- 批次
- 模型
- 状态
- 问题数
- 评分
- 创建时间
- 完成时间

命名建议：

- `review-list-2026-04-17-153000.xlsx`

### 7.2 导出报告

`导出报告` 面向“完整评审结果内容”的批量导出。

导出格式为 `zip`，每条可导出的任务生成一个 `md` 文件。

可导出条件统一定义为：

- `review.reportMarkdown` 存在且非空

这意味着：

- 已完成任务通常可导出
- 部分完成任务若有报告正文，可导出
- 失败任务若有报告正文，可导出
- 未完成任务默认被跳过
- 没有 `reportMarkdown` 的任务一律被跳过

命名建议：

- zip 文件：`review-reports-2026-04-17-153000.zip`
- zip 内单文件：`玩法复盘__玩法.xlsx__completed.md`

文件名生成时需要做安全字符清洗，避免路径分隔符与保留字符导致下载异常。

### 7.3 导出反馈

导出完成后，前端要给出简短结果反馈。

示例：

- `已导出 8 份报告`
- `已导出 8 份报告，跳过 3 条未完成任务`
- `未找到可导出的报告`

不需要在本轮引入复杂通知中心，局部消息反馈即可。

## 8. 技术设计

### 8.1 前端改动

主要修改 `components/review-jobs-table.tsx`：

- 新增勾选列
- 增加 `selectedIds` 与 `allMatchingSelected` 状态
- 渲染批量操作条
- 处理：
  - 单行勾选
  - 表头全选
  - 全选筛选结果
  - 清除选择
  - 触发导出
  - 打开删除确认弹窗

同时新增一个轻量确认对话框组件，用于批量删除二次确认。组件只需满足本轮：

- 标题
- 正文
- 可选摘要
- 确认 / 取消

### 8.2 服务端接口

保留现有：

- `GET /api/reviews`

新增：

- `POST /api/reviews/export-list`
- `POST /api/reviews/export-report`
- `DELETE /api/reviews`

三者都使用统一的批量范围请求结构：

- `selectedIds?: string[]`
- `query?: string`
- `allMatching?: boolean`

服务端收到请求后，统一解析出实际任务集合。

### 8.3 领域服务拆分

建议在 `lib/` 下围绕 `review-jobs` 继续拆出可测试 helper：

- `resolveReviewSelectionScope`
  - 解析批量作用范围
- `deleteReviewJobs`
  - 执行删除
- `exportReviewListWorkbook`
  - 生成清单导出数据与文件
- `exportReviewReportArchive`
  - 生成报告 zip 和跳过统计
- `canExportReviewReport`
  - 统一判断报告是否可导出

其中 `canExportReviewReport` 的实现应只看 `reportMarkdown`，不要把状态判断分散在多个调用点。

### 8.4 文件生成

项目已经依赖 `xlsx`，因此清单导出可直接使用现有依赖生成。

报告 zip 需要新增压缩依赖或使用运行时可接受的压缩方案。此处不要求复用 Electron 专属能力，应保持 Web 与桌面壳下都能工作。

## 9. 错误处理

### 9.1 删除失败

- 删除接口失败时，保留当前选择
- 在批量条附近展示简短错误提示
- 不自动清空选择，便于重试

### 9.2 导出失败

- 导出接口失败时，展示简短错误提示
- 不清空选择
- 不把“部分导出成功”伪装成完全失败；如服务端已成功生成部分结果，应明确反馈导出数与跳过数

### 9.3 空作用范围

如果用户没有选择任何项就点击批量按钮：

- 前端应直接禁用按钮
- 服务端仍需兜底校验，返回明确错误，避免空请求执行

## 10. 测试设计

### 10.1 组件测试

扩展 `tests/components/review-jobs-table.test.tsx`，覆盖：

- 搜索后仍可勾选
- 勾选任意行后出现批量条
- 表头全选作用于当前筛选结果
- 从“已勾选”切换到“筛选结果全部”
- 清除选择后批量条消失
- 删除确认文案在普通状态和包含进行中状态之间切换

### 10.2 服务测试

围绕选择范围与导出规则补单测，覆盖：

- `selectedIds` 范围解析
- `query + allMatching` 范围解析
- 报告导出跳过无 `reportMarkdown` 的任务
- 删除任务时只删除 `ReviewJob` 与 `Annotation`，不删除 `Document`

### 10.3 接口测试

至少覆盖：

- 导出清单成功
- 导出报告成功
- 导出报告时存在跳过项
- 删除成功
- 空选择请求被拒绝

## 11. 验收标准

- 用户可以在 `评审列表` 中勾选多条任务并执行批量删除
- 用户可以把当前筛选结果全部纳入批量范围
- 所有删除都要求二次确认
- 包含进行中任务时使用更强确认文案
- 用户可以批量导出 `xlsx` 清单
- 用户可以批量导出 `zip` 报告包
- 导出报告时会自动跳过不可导出任务，并给出明确反馈
- 原有搜索、刷新、查看详情流程不被破坏

## 12. 后续扩展空间

本设计完成后，后续可在同一结构上继续扩展：

- 单行删除
- 批量重试
- 导出格式选择器
- 分页与跨页选择
- 批量操作结果通知中心

这些能力都不需要推翻本轮的“选择态批量操作条 + 统一范围解析服务”结构。
